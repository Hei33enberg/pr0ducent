# AGENTS.md

## Learned User Preferences

- Ship changes by committing and pushing to `main` (pull/rebase onto `origin/main` when the branch has diverged) so connected tools such as Lovable see updates; use GitHub identity aligned with `maciej.bialek.33@gmail.com` for this repo’s commits when working on behalf of this project owner.
- Keep product messaging aligned with what the system actually does: separate live builder integrations (e.g. v0) from benchmark-only tools, and present post-build benchmarking and PVI-style scoring as a first-class part of the product story—not only as placeholders or mocks.
- When the owner will not run a separate Supabase dashboard or repeated manual steps, prefer automatable delivery (CI, provision scripts, one-time GitHub secrets), Lovable operator steps (Pull / build / Publish), **pasteable deploy checklists** for Lovable that name migrations, GRANT expectations, Edge functions to redeploy, and smoke steps, and plain-language architecture explanations over runbooks that assume they will click through Supabase or Lovable for every release. When handing work to Lovable or Antigravity (AG), give **ready-to-paste prompts and concrete next steps** (including migrations/GRANT/Edge/smoke when relevant) instead of asking the owner to choose options when the path is already clear.
- When designing scale for many concurrent runs, plan explicitly for durable workflow engines and broker account isolation as layers beyond the first Supabase Edge–centric MVP if they are not already in place.
- In Supabase Edge Functions that use `@supabase/supabase-js` with the caller’s JWT, validate identity with supported APIs such as `auth.getUser()` (using the Bearer token or client global headers); do not rely on non-existent or undocumented client methods.
- Prefer sequencing that ships broker-mode orchestration and monetization on the current stack first, then infra moves (e.g. Vercel + direct Supabase); BYOA user keys (Vault + typed RPCs) ship as an explicit slice on the same stack when prioritized, not only as a vague future phase.

## Learned Workspace Facts

- Frontend stack is React 18, TypeScript, Vite, Tailwind, shadcn/ui; routing is lazy-loaded in `src/App.tsx`; i18n uses `src/lib/i18n.tsx` with `src/locales/en.json` and `src/locales/pl.json`. **MVP:** `I18N_SINGLE_LOCALE_ENGLISH` in `i18n.tsx` locks the UI to English (no nav/footer language toggles); set it `false` and restore toggles when shipping Polish. **Newsc0rp design system:** shared brand/tokens with murd0ch and p0xi; do not maintain a second full brandbook here—follow `docs/NEWSCORP-DESIGN-SYSTEM.md` and edit the master in `newsc0rp-main` (`src/styles/design-tokens.md`, §15 multi-product) when changing shared LP rules. **Shared shell (header/hero conventions):** `docs/CROSS-PRODUCT-SHELL.md` — keep `PageFrame` / header metrics aligned with `newsc0rp-main` `Index.tsx` where the product does not differ. **LP layout:** blocks below the hero (e.g. `ExperimentHistory`) should use the same horizontal padding rhythm as `HeroSection` (`px-4 sm:px-6 md:px-8 lg:px-12`) and matching `max-w-*` centering so sections do not look inset wrong. **Past experiments on the homepage:** render that block only when there is at least one saved run (`pastExperiments.length > 0`); guests use local history, signed-in users use DB-backed lists. **LP before backend:** `docs/LP-CHECKLIST.md` — smoke, prompt-first vs story-first murd0ch, content/crons, deploy.
- Supabase powers Postgres, Auth, and Edge Functions under `supabase/functions/`; logged-in multi-builder dispatch goes through `dispatch-builders` (and `process-task-queue` for queued work) with `run_jobs`, `run_tasks`, `run_events`, and `builder_integration_config` (see `docs/ORCHESTRATOR.md`). BYOA: user keys are stored via Vault-backed RPCs (`save_user_builder_api_key`, `disconnect_user_builder_api_key` for authenticated clients; `get_byoa_api_key_for_dispatch` for service role in Edge); live dispatches can emit `orchestrator.credential_source` in `run_events` for audit. Guests use `run-on-v0` outside that queue.
- Operational and handoff documentation includes `docs/DEVELOPMENT-STATUS.md`, `docs/SPRINT-CLOSE.md`, `docs/RAG-PHASE2.md`, `docs/BYOA-MIGRATION.md`, `docs/LOVABLE-DEPLOY-BYOA-SPRINT-C.md`, `docs/VERCEL-SUPABASE-MIGRATION.md`, and `docs/WORKFLOW-ENGINE.md`.
- Repository `.gitignore` excludes `.cursor`; avoid committing editor state or stray nested duplicate project trees (e.g. an extra top-level `pr0ducent/` folder). **License:** the codebase is **MIT** ([LICENSE](./LICENSE), [docs/POP-OSS-SCOPE.md](./docs/POP-OSS-SCOPE.md)); forks are expected — never commit production secrets.
- `supabase/config.toml` sets `verify_jwt` per function: for example `dispatch-builders` and `score-builder-output` expect user JWT; `poll-v0-status` is typically invoked with anon from the browser (`verify_jwt = false`).
- Typical split: Cursor drives the GitHub repo (migrations, Edge Functions, backend-heavy changes); Antigravity ships UI commits to `main`. **Primary hosting target:** Vercel + Supabase under the team’s own accounts (`docs/VERCEL-SUPABASE-MIGRATION.md`); **Lovable** is optional for fast LP work (e.g. murd0ch) and must merge to `main` — official app deploys from GitHub to Vercel, not only via Lovable Publish.
- **VBP (Vibecoding Broker Protocol)** is the sole public name for the third-party builder integration standard (telemetry, exports, partner APIs); benchmarking and scoring of built outputs are treated as core to the pr0duction pipeline alongside dispatch, not as an afterthought. See `docs/VBP-POP-BRANDING.md`.
- If the hosted Supabase schema (e.g. Lovable-linked) is behind `main`, apply migrations from this repo instead of stripping Edge logic (circuit breaker, `next_retry_at`, VBP columns) to match an outdated live database.
- Avoid `as any` in `src/` for app code; use `Database` / `Json` from `src/integrations/supabase/types.ts` (including registered RPCs) and `satisfies` where introduced; BYOA JSON-RPC handling helpers live in `src/lib/byoa-rpc.ts`. PVI for benchmarks: `src/lib/pvi-engine.ts` and `PVIDimensions` / `score_*` in `builder_benchmark_scores`; marketing pricing UI uses `pvi-calculator.ts` only.
- Read-only UI contracts: `WinnerBanner` uses `avg_pvi` from `builder_leaderboard`; radar-style charts use `score_*` from `builder_benchmark_scores`. Changing the MV or columns requires a coordinated front + migration release. Map realtime `BuilderResultRow` (snake_case) to `BuilderResult` (camelCase) before props that expect `BuilderResult`.
- Runtime catalog: `BuilderCatalogProvider` loads `builder_integration_config` and merges with defaults in `src/config/tools.ts` (`mergeBuilderCatalog`). `useBuildersRegistry` is an alias of `useBuilderCatalog`—one implementation, two names for docs compatibility.
- Non-v0 completion: `poll-builder-status` (JWT) uses `poll_url_template` / VBP status URL and `poll_failed_values`; `pbp-webhook` updates `builder_results` and `run_tasks` for lifecycle events; Group B/C jobs update scores by `builder_result_id` with idempotent writes and `run_events` for audit.
